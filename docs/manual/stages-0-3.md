# Manual testing guide: Этапы 0–3

Полный прогон ручного теста от чистого развёртывания до полного чтения фика.
Проходится в 2 аккаунтах Telegram:

- **A (admin)** — будет модератором/админом. Его `tg_id` должен быть в `ADMIN_TG_IDS`.
- **B (author)** — обычный пользователь, который напишет и опубликует фик, а потом
  будет его читателем (через 3-й аккаунт) — либо можно просто читать своим же.
- **C (reader)** — ещё один аккаунт для проверки навигации и чтения. Можно
  пропустить и использовать A для чтения (A и B тогда на разных устройствах).

В нём 5 разделов:

1. [Этап 0 · развёртывание](#этап-0--развёртывание)
2. [Этап 1 · онбординг и роли](#этап-1--онбординг-и-роли)
3. [Этап 2 · авторство и модерация](#этап-2--авторство-и-модерация)
4. [Этап 3 · читалка, каталог, полка](#этап-3--читалка-каталог-полка)
5. [Сквозная проверка навигации](#сквозная-проверка-навигации)

Для каждой проверки указано: **что делать**, **что ожидать**, и **как убедиться**
(SQL/лог).

## Этап 0 · развёртывание

### 0.1 Первый запуск стэка

```bash
cp .env.example .env
# Отредактировать .env: BOT_TOKEN, POSTGRES_PASSWORD, MEILI_MASTER_KEY, ADMIN_TG_IDS
make build
make migrate
make up
make logs
```

**Ожидание:**

- Поднимаются контейнеры: `bot`, `worker`, `scheduler`, `postgres`, `redis`, `meilisearch`.
- В логах `bot_init env=dev`, `bot_starting_polling` — бот ждёт апдейты.
- В логах `worker` и `scheduler` — процесс жив.
- `http://localhost:<HEALTH_PORT>/healthz` → `{"status":"ok"}`.
- `http://localhost:<HEALTH_PORT>/metrics` → prometheus-формат.

### 0.2 Миграции и схема

```bash
make db-shell
```

В psql:
```sql
\dt
\d fanfics
\d chapters
\d chapter_pages
\d bookmarks
\d likes
\d reads_completed
\d reading_progress
\d outbox
```

**Ожидание:** все перечисленные таблицы существуют со столбцами, индексами и FK.
Специально проверить:

- `fanfics.likes_count`, `fanfics.reads_completed_count` — есть (счётчики).
- `chapter_pages.entities JSONB` — есть.
- `reading_progress` PRIMARY KEY `(user_id, fic_id)` — один курсор на фик.
- `outbox.published_at` — nullable timestamptz (используется диспетчером).

### 0.3 Запуск тестов

```bash
make test-unit   # или:
.venv/bin/pytest tests/unit/ -q --ignore=tests/unit/test_config.py
```

**Ожидание:** **134 passed** (из них 12 паджинатор-тестов hypothesis + 11 Reading
use case'ов).

## Этап 1 · онбординг и роли

### 1.1 Первый `/start` для пользователя B

Аккаунт B в боте: `/start`.

**Ожидание:**

1. Пришло welcome-сообщение ("Привет! 📚 Это Fantik...").
2. Следом — правила с кнопкой «✅ Принимаю правила».
3. В БД: `SELECT id, username, first_name, agreed_at, role, created_at FROM users WHERE id = <tg_id>;`
   - Запись создана.
   - `agreed_at IS NULL`.
   - `role = 'user'`.

### 1.2 Принятие правил

B нажимает «✅ Принимаю правила».

**Ожидание:**

- Ответ «Отлично! Добро пожаловать в Fantik.» + главное меню с кнопками
  «📚 Каталог», «📖 Моя полка», «✍️ Стать автором», «👤 Профиль».
- В БД: `agreed_at IS NOT NULL`.

### 1.3 Повторный `/start`

B ещё раз `/start`.

**Ожидание:** приходит «С возвращением! 📚 Куда дальше?» + главное меню. Новая
запись в users **не** создаётся (идемпотентность).

### 1.4 UTM-трекинг

B нажимает STOP командой (т.е. удаляет чат) и открывает бота по ссылке
`https://t.me/<bot>?start=TESTCODE1`.

**Ожидание:**

- Пользователь «приходит заново» с utm_code = `TESTCODE1`.
- В БД: `SELECT utm_source_code_id FROM users WHERE id = <B_tg_id>;` — заполнено
  (first-touch атрибуция, не перезаписывается).
- `SELECT * FROM tracking_events WHERE user_id = <B_tg_id>;` — событие start
  записано с `utm_code_id`.

Повторный открытие по ДРУГОМУ коду НЕ должен изменять `users.utm_source_code_id`
(first-touch). Это фиксируется в новом `tracking_events`, но не в users.

### 1.5 Установка ника автора

B → «👤 Профиль» → «✍️ Стать автором» (кнопка). Ввести ник `mark_the_writer`.

**Ожидание:**

- «Ник **mark_the_writer** закреплён за тобой. Теперь можно создавать работы.»
- В меню появляются «✍️ Мои работы», «➕ Новая работа».
- В БД: `SELECT author_nick FROM users WHERE id = <B_tg_id>;` → `mark_the_writer`.

**Валидация ника (негативные кейсы):**

- `m` — отклоняется (< 2 символов).
- `Mark.Writer` — отклоняется (точка запрещена).
- `mark the writer` — отклоняется (пробел).
- Другой юзер пробует `mark_the_writer` — «Этот ник уже занят.» (UNIQUE по LOWER).

### 1.6 Админская роль (seed через ADMIN_TG_IDS)

В `.env`: `ADMIN_TG_IDS=<tg_id аккаунта A>`. Переподнять бот (`make down && make up`).
Аккаунт A → `/start` (если ранее не делал) → принять правила.

**Ожидание:** В главном меню A появляются «🛡 Модерация» и «⚙️ Админ».
В БД: `SELECT role FROM users WHERE id = <A_tg_id>;` → `admin`.

## Этап 2 · авторство и модерация

### 2.1 Создание черновика фика

B → «➕ Новая работа» → ввести название → ввести summary с форматированием
(bold, italic, spoiler, ссылка) → выбрать фэндом → выбрать age rating → теги →
опционально обложка → завершить.

**Ожидание после создания:**

- Карточка черновика показывается: название, summary, fandom, rating, теги,
  обложка (если была), «📖 Глав: 0», «Статус: DRAFT».
- В БД: `SELECT id, title, status, fandom_id, summary FROM fanfics ORDER BY id DESC LIMIT 1;`
  - `status = 'draft'`.
  - `summary_entities` — валидный JSON массив с entities.
  - `cover_file_id` — `NULL` или valid file_id.

### 2.2 Добавление глав

B → карточка фика → «➕ Глава» → ввести заголовок → ввести текст главы с
форматированием (bold, italic, code, spoiler, emoji 😀, emoji с ZWJ 👨‍👩‍👧).

**Кейсы:**

- Короткая глава (200 символов) — добавляется, `chapters_count = 1`.
- Длинная глава (~10 000 UTF-16 units, > 3 страниц после пагинации) — добавляется.
- Попытка ввести текст > 100 000 UTF-16 units — отклоняется с ошибкой
  «Превышен лимит UTF-16 units в главе».

Добавить ещё одну главу, чтобы фик имел 2 главы.

**Ожидание:**

- В БД: `SELECT number, title, chars_count, status FROM chapters WHERE fic_id = <fic_id> ORDER BY number;`
  - 2 строки, `status = 'draft'`, `chars_count` корректно в UTF-16.
  - `entities` валидный JSON.
- `SELECT chapters_count, chars_count FROM fanfics WHERE id = <fic_id>;` — cчётчики
  инкрементированы.

### 2.3 Правка черновика

B → карточка фика → «✏️ Название» → изменить. Проверить, что summary тоже можно
править. Главы: «✏️ Глава 1» → поменять текст.

**Ожидание:**

- Все правки применяются только в draft/rejected/revising (в approved не даст).
- `updated_at` в БД растёт.

### 2.4 Отправка на модерацию

B → карточка фика → «📨 На модерацию».

**Ожидание:**

- «Фик отправлен на модерацию (случай #N).»
- В БД:
  - `fanfics.status = 'pending'`.
  - `SELECT * FROM fanfic_versions WHERE fic_id = <fic_id>;` — создан снапшот.
  - `SELECT * FROM moderation_queue WHERE fic_id = <fic_id>;` — создан case,
    `decision IS NULL`, `submitted_by = <B_tg_id>`.
  - Все главы `status = 'pending'`.

### 2.5 Дневной лимит подач

B быстро создаёт и отправляет на модерацию 11 фиков подряд (дефолт 10). 11-й
должен получить ошибку «Дневной лимит подач исчерпан».

*(Можно пропустить — долго; настройка `max_fics_per_day`.)*

### 2.6 Очередь модерации

Аккаунт A → «🛡 Модерация» → «🎯 Следующая работа».

**Ожидание:**

- Карточка с фикой B: название, fandom, rating, теги, ссылка на автора, кнопки
  «📖 Глава 1», «📖 Глава 2», «✅ Одобрить», «❌ Отклонить», «🎯 Следующая».
- Case в БД заблокирован на A: `locked_by = <A_tg_id>`, `locked_until` — +15 мин.

### 2.7 Self-moderation блок

B (как обычный юзер) не видит «🛡 Модерация». Если каким-то образом B получит
admin role и попытается одобрить СВОЙ же фик — `ApproveUseCase` кинет
`ForbiddenActionError("Нельзя модерировать свои работы.")`. Проверяется unit-тестом
`test_cannot_approve_own`.

### 2.8 Просмотр глав в карточке модерации

A → клик «📖 Глава 1» → получает заголовок «📖 Глава 1. <название>» и отдельным
сообщением сырой текст с entities (форматирование сохранено).

### 2.9 Approve

A → «✅ Одобрить».

**Ожидание:**

- «Решение применено.» + меню модерации.
- Карточка в чате деактивируется (клавиатура убирается).
- Пользователь B получает личное сообщение: «Твой фик "..." одобрен. ✅»
- В БД:
  - `fanfics.status = 'approved'`, `first_published_at IS NOT NULL`,
    `current_version_id` ссылается на снапшот.
  - Все главы `status = 'approved'`.
  - `moderation_queue.decision = 'approved'`, `decided_by = <A_tg_id>`.
  - `SELECT event_type, payload->>'fic_id', payload->'chapter_ids', published_at FROM outbox ORDER BY id DESC LIMIT 1;`
    - `event_type = 'fanfic.approved'`, `chapter_ids` — массив id глав.
    - `published_at IS NULL` (диспетчер ещё не обработал).
- `SELECT * FROM audit_log WHERE target_id = <fic_id> ORDER BY id DESC LIMIT 1;` —
  запись `action='fic.approve'`.

### 2.10 Через ≤ 60 секунд — репагинация

**Ожидание:**

- `SELECT published_at FROM outbox WHERE id=<outbox_id>;` → заполнилось (диспетчер
  отработал).
- `SELECT count(*), max(page_no) FROM chapter_pages WHERE chapter_id = <ch_id>;` —
  количество страниц совпадает с ожидаемым для длины главы (для 10k units ≈ 3 страницы).

Если не произошло за 60 сек — проверить логи `worker`: `outbox_dispatch_tick_done processed=N` + `repaginate_chapter_done chapter_id=X pages=N`.

### 2.11 Reject flow

B ещё раз отправляет черновик на модерацию. A → «❌ Отклонить» → выбрать
причины (чекбоксы, минимум 1) → «Далее» → ввести комментарий с форматированием
→ «Подтвердить».

**Ожидание:**

- B получает сообщение «Твой фик "..." отклонён. Причины: ... Комментарий: ...» —
  форматирование комментария сохранено (entities).
- `fanfics.status = 'rejected'`.
- `moderation_queue.decision = 'rejected'`, `reasons = [id1, id2, ...]`.

### 2.12 Revise flow

B → карточка фика → «✏️ Доработать» (доступна в rejected).

**Ожидание:**

- `fanfics.status = 'revising'`.
- Можно править → отправить на модерацию снова.

### 2.13 Cancel submission

B отправляет ещё один черновик на модерацию. До того как модератор взял в
работу, B → «❌ Отменить отправку».

**Ожидание:**

- `moderation_queue.cancelled_at IS NOT NULL`.
- `fanfics.status` вернулся в `draft`.
- A → «🎯 Следующая» не получит этот case (отфильтрован по `cancelled_at`).

### 2.14 Concurrency: два модератора и `FOR UPDATE SKIP LOCKED`

*(Если есть второй модератор.)* A1 и A2 одновременно жмут «🎯 Следующая» для
одной и той же очереди из 2 cases. Каждый получит РАЗНЫЙ case (SKIP LOCKED).
Проверяется интеграционным тестом `test_pick_next_skip_locked`.

### 2.15 Stale lock

A берёт в работу case → закрывает чат на 20 минут. Другой модератор A2
→ «🎯 Следующая» получит тот же case (lock истёк). Или админ `/release_stale_locks`.

### 2.16 Правка approved-фика → всё в pending

B → approved фик → «✏️ Название» → изменить.

**Ожидание:**

- `fanfics.status = 'pending'` (обратно).
- Все главы: `status = 'pending'`.
- Новый `fanfic_versions` снапшот.
- Новая запись в `moderation_queue`.

После approve от A:
- `chapter_ids` в payload `fanfic.approved` — все главы.
- Воркер репагинирует все главы.
- В `chapter_pages` старые строки удалены, новые вставлены.

## Этап 3 · читалка, каталог, полка

### 3.1 Открытие каталога

Аккаунт C (или B, или A) → `/catalog` или «📚 Каталог» из главного меню.

**Ожидание:**

- «📚 Каталог работ» + клавиатура:
  - Ряд 1: «🆕 Новое», «🔥 Топ»
  - Ряд 2: «🏷 По фэндому»
  - Ряд 3: «← Главное меню»

### 3.2 Лента «Новое»

C → «🆕 Новое».

**Ожидание:**

- Заголовок «🆕 Новое».
- Список фиков: `1. <title> — <author_nick> · ❤️ <likes> · 📖 <chapters> гл.`
- По 10 штук на страницу.
- Пагинация: ◀ / `стр. N` / ▶.
- Внизу: «⟵ Каталог».

### 3.3 Лента «Топ»

C → «🔥 Топ».

**Ожидание:** Заголовок «🔥 Топ». Сортировка по `likes_count DESC`.

### 3.4 Фильтр по фэндому

C → «⟵ Каталог» → «🏷 По фэндому» → список фэндомов (по 12 на страницу,
пагинация) → выбрать фэндом.

**Ожидание:**

- Заголовок «🆕 Новое · <fandom_name>».
- В ленте только фики выбранного фэндома.

### 3.5 Открытие фика из каталога

C → в ленте нажимает на название фика.

**Ожидание:**

- Отправляется НОВОЕ сообщение — **photo** (cover) с подписью: «<b>title</b> · <author_nick>\n\n<summary>».
- Форматирование summary (entities) — сохранено.
- Клавиатура:
  - «📖 Читать» (если прогресса нет)
  - «⟵ Каталог»
- Если у фика нет обложки — вместо photo отправляется обычное текстовое
  сообщение с тем же содержимым.

### 3.6 Начало чтения

C → «📖 Читать».

**Ожидание:**

- Cover-сообщение **удаляется**.
- Новое текстовое сообщение:
  ```
  Глава 1 · «<title>»           ← bold
  Страница 1 из N                ← italic

  <текст страницы с entities>
  ```
- Клавиатура (reader_kb):
  - Ряд 1: ◀ Назад (серый) · 📄 1/N · Дальше ▶
  - Ряд 2: ⏮ Глава (серый) · 📖 Оглавление · Глава ⏭ (активна, если есть глава 2)
  - Ряд 3: 🔖 · 🤍 · ⚠️
  - Ряд 4: «⟵ Каталог»

### 3.7 Листание «Дальше»

C → «Дальше ▶».

**Ожидание:**

- **Edit** того же сообщения (не новое!) на страницу 2.
- Счётчик «📄 2/N» обновился.
- Форматирование на странице 2 сохранено.
- Ни одна entity не обрезана посередине. Emoji/ZWJ не сплитится.
- `SELECT * FROM reading_progress WHERE user_id = <C_tg_id>;` → запись с
  `chapter_id = <ch1>`, `page_no = 2` (throttle write прошёл).

### 3.8 Двойной клик ▶

C → быстро двойной клик «Дальше».

**Ожидание:**

- Никаких ошибок в UI. Алерта нет.
- В логах может быть `reader_edit_failed` c «message is not modified» — это
  подавляется.
- `reading_progress.page_no` не меняется рывком; throttle 5 сек защищает.

### 3.9 Листание «Назад»

C → «◀ Назад».

**Ожидание:** страница N-1. Форматирование сохраняется.

### 3.10 Переход между главами

C на последней странице главы 1 → «Глава ⏭» (справа от 📄 N/N).

**Ожидание:**

- Edit на «Глава 2 · <title>\nСтраница 1 из M».
- Клавиатура: «⏮ Глава» уже активна.

### 3.11 Оглавление

C → «📖 Оглавление».

**Ожидание:**

- **Edit** только клавиатуры (текст страницы остаётся).
- Вместо reader_kb — список глав: ▶ перед текущей.

### 3.12 Клик по главе из оглавления

C → «▶ Глава 1. <title>».

**Ожидание:**

- Edit текста на «Глава 1, стр. 1».
- Клавиатура снова reader_kb.

### 3.13 Закладка

C → 🔖 → становится 📑; alert «Добавлено в закладки».

**Ожидание:**

- `SELECT * FROM bookmarks WHERE user_id=<C_tg_id>;` → 1 запись.
- Иконка в клавиатуре меняется на 📑.

Повтор: 📑 → 🔖; alert «Убрано из закладок». Запись удалена.

### 3.14 Лайк + атомарный счётчик

C → 🤍 → становится ❤️; alert «❤️ Добавлен лайк».

**Ожидание:**

- `SELECT * FROM likes WHERE user_id=<C_tg_id>;` → 1 запись.
- `SELECT likes_count FROM fanfics WHERE id=<fic_id>;` — выросло на 1.

Повтор: ❤️ → 🤍. Запись удалена, счётчик `GREATEST(likes_count-1, 0)` — не уйдёт
в минус, даже при повторных кликах.

### 3.15 Report-заглушка

C → ⚠️.

**Ожидание:** alert «Жалобы появятся в следующих обновлениях.» Ничего в БД не
меняется.

### 3.16 Дочитано

C → долистывает до последней страницы последней главы.

**Ожидание:**

- В reader_kb появляется дополнительный ряд «✓ Дочитано».
- Обычные прошлые ряды остаются.

C → «✓ Дочитано».

**Ожидание:**

- Alert «✓ Дочитано!».
- `SELECT * FROM reads_completed WHERE user_id=<C_tg_id>;` → запись с
  `chapter_id = <last_chapter_id>`.
- `SELECT reads_completed_count FROM fanfics WHERE id=<fic_id>;` — выросло на 1.
- В outbox новое событие `fanfic.read_completed`.

**Идемпотентность:** повторный клик «✓ Дочитано» — alert «Отмечено.» (не
«Дочитано!»); счётчик `reads_completed_count` НЕ увеличивается. Проверяется
unit-тестом `test_last_chapter_increments_fic_counter_once`.

### 3.17 Прогресс чтения — «Продолжить»

C → закрывает чат → открывает бот → `/catalog` → выбирает тот же фик.

**Ожидание:**

- Cover с двумя кнопками:
  - «▶ Продолжить (гл.X, стр.Y)» — последняя позиция из `reading_progress`.
  - «📖 С начала».
- Клик «Продолжить» → cover удаляется → отправляется текст с гл.X стр.Y.
  (Регрессия Bug #3 — не `edit_message_text` на photo.)

## Сквозная проверка навигации

Проверяем, что юзер **нигде не застревает** и всегда может вернуться в меню.

| Откуда | Нажимаем | Куда приходим |
|---|---|---|
| Главное меню | «📚 Каталог» | Каталог root |
| Каталог root | «🆕 Новое» | Лента «Новое» |
| Лента «Новое» | ◀ / ▶ | Предыдущая / следующая страница ленты |
| Лента «Новое» | «⟵ Каталог» | Каталог root |
| Каталог root | «← Главное меню» | Главное меню |
| Лента «Новое» | Клик по фику | Cover фика (new message) |
| Cover | «📖 Читать» | Страница 1 (cover удалён) |
| Cover | «⟵ Каталог» | Каталог root (new message, cover остаётся) |
| Чтение | «◀ Назад» / «Дальше ▶» | Предыдущая / следующая страница (edit) |
| Чтение | «⏮ Глава» / «Глава ⏭» | Первая страница соседней главы (edit) |
| Чтение | «📖 Оглавление» | Список глав (edit клавиатуры) |
| Оглавление | Клик по главе | Её стр.1 (edit) |
| Чтение (любая стр.) | «⟵ Каталог» | Каталог root (edit текста) |
| Чтение (последняя стр.) | «✓ Дочитано» | Alert + клавиатура без кнопки «Дочитано» |
| Главное меню | «📖 Моя полка» | Полка: вкладка «Недавно» |
| Полка | «📑 Закладки» / «❤️ Лайки» | Другая вкладка (edit) |
| Полка | Клик по фику | Cover фика |
| Полка | «← Главное меню» | Главное меню |
| Главное меню | «✍️ Мои работы» | Список моих фиков |
| Список моих фиков | Клик по фику | Карточка автора |
| Карточка автора | «← Мои работы» / «← Главное меню» | Возврат |

**Проверить, что НЕ ломается:**

- Если кликать на «⟵ Каталог» когда текущее сообщение — текст чтения,
  `edit_text` работает → каталог отобразится вместо страницы.
- Если кликать «⟵ Каталог» когда текущее сообщение — photo (cover),
  `edit_text` упадёт → но browse router ловит исключение и шлёт **новое
  сообщение**. Пользователь видит каталог, старый cover остаётся в истории.
- Двойной клик по любой навигационной кнопке: alert «message is not modified»
  — подавляется, пользователь ничего не видит.

## Прогон полного сценария (happy path end-to-end)

1. **A, B, C** делают `/start`, принимают правила.
2. **B** задаёт ник, создаёт фик с 2 главами (первая короткая, вторая длинная ~10k units с emoji+bold+spoiler+code), добавляет обложку, отправляет на модерацию.
3. **A** принимает фик. Через ≤60 сек в `chapter_pages` появляются строки.
4. **C** открывает `/catalog → Новое → <фик>` → видит cover с «📖 Читать».
5. **C** читает страницу 1 → 2 → 3 (вторая глава), ставит ❤️ и 📑. В БД `likes_count=1`, `bookmarks=1 запись`.
6. **C** долистывает до последней страницы, жмёт «✓ Дочитано». В БД `reads_completed_count=1`, `reads_completed=1 запись`.
7. **C** возвращается в `/shelf` → Недавно содержит фик со стр. последней.
8. **B** правит текст главы 1, отправляет на повторную модерацию.
9. **A** одобряет → воркер репагинирует главу 1 → в `chapter_pages` новые строки.
10. **C** снова открывает фик → cover показывает «▶ Продолжить (гл.2, стр.N)» (его прогресс) и «📖 С начала». Клик «Продолжить» → корректно открывает нужную страницу.
11. **C** → «← Главное меню» → главное меню.

На каждом шаге — **никаких** ошибок в логах бота; счётчики в БД согласованы;
форматирование сохраняется.

## Важные известные ограничения

- Авто-фиксация «дочитано через 30 сек просмотра» — не реализовано (только кнопка).
- Префетч страницы N+1 — no-op (будет в Этапе 7 через отдельный dishka-scope).
- Жалобы (⚠️) — заглушка до Этапа 5.
- Поиск по фикам — Этап 4 (Meilisearch).
- Если прогресс-throttle 5 сек блокирует запись последней просмотренной страницы,
  эта запись теряется (не планируется delayed write в Stage 3).
